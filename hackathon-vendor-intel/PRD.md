# PRD｜厂商情报与运营台（精简版）

负责人：黄士鑫　|　版本：v0.1（黑客松周末冻结）　|　日期：2026-06-01

## 1. 背景
TikTok GIP 是达人营销产品，撮合游戏厂商与 TikTok 创作者：**厂商付钱，达人创作内容，实现游戏社媒营销，丰富游戏内容生态**。其中「厂商运营」负责对接厂商、了解需求、给诊断和经营建议。今天他们的情报和诊断高度依赖人工，散在各处、难成体系。

## 2. 目标
给厂商运营一个 Web 工作台：**输入一个游戏，就能看它最近有什么动作、在我们平台做得怎么样、GIP 花了多少钱。** 把「发现 → 诊断 → 看消耗」串成一条链路，降低运营的信息收集和判断成本。

## 3. 范围（本期只做三块，智能体后续再说）

**① 情报抓取（情报雷达）**　自动盯市场，告诉运营最近有哪些新游、哪些游戏出了版本更新、谁有值得关注的大动作。是整个台子的入口，先解决「我该关注谁」。产出 `GameMove`。

**② 平台诊断报告**　运营选中一个游戏，一键出它现阶段在平台的诊断报告，含三块：平台生态（数据 + 案例）、竞品对比、推荐营销动作。需额外输出**业务 context** 来训练判断标准（怎么算「做得好」、推荐动作有哪些套路），否则诊断没有业务含金量。产出 `DiagnosisReport`。

**③ GIP 数据库（预算消耗面板）**　分时段 / 游戏 / 品类 / 厂商看 GIP 预算与消耗；可搜游戏、下钻到某游戏的活动消耗明细。需整理数据源。产出 `GIPRecord`。

## 4. Demo 动线（必须连起来）
首页搜索框输入游戏 → 情报雷达发现一个有动作的游戏 → 选中 → 出诊断报告 → 跳 GIP 面板看预算消耗。一个游戏从头到尾跑通，演示即成立。

## 5. 形态与技术栈
一个 Web 网页台：顶部搜索框打底，三个面板对应三件事。Next.js 14 App Router + TypeScript，Tailwind + shadcn/ui，数据走飞书多维表格 + Mock，LLM 封装在 `lib/llm.ts`，Vercel 一键部署。

## 6. 架构原则（5 人 1.5 天能拼起来的关键）
- **契约先行**：先冻结 `types/contract.ts`，所有人对着同一份契约和 mock 写。
- **文件级隔离**：单仓库、一人一目录，合并不打架。
- **Mock 优先**：API 先返回符合契约的 mock，前端对着 mock 开发，Demo 永远能跑。
