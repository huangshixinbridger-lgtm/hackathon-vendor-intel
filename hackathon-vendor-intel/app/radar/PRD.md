# PRD｜模块① 情报雷达

> 版本：v2.0  
> 更新日期：2026-06-04  
> 所属项目：情报与运营台  
> 模块 Owner：Gardner  
> 当前实现目录：`app/radar`

## 1. 产品定位

情报雷达是「情报与运营台」的发现入口，负责沉淀游戏厂商、游戏项目和游戏动态三类数据，并将市场动态整理为符合项目契约的 `GameMove` 输出，供其他模块调用。

本模块回答厂商运营每天最关心的问题：

> 最近哪些游戏或厂商有动作？今天新增了多少动态、涉及哪些游戏和厂商？我应该优先关注哪些情报？

本模块只聚焦游戏厂商、游戏项目和市场动态，不做 TikTok 内容生态诊断、达人分析、GIP 消耗分析，也不在页面中承载「看雷达 / 去诊断 / 查 GIP」等跨模块动作。

## 2. 本期目标

1. 建立并展示三张核心库表：游戏动态表、游戏项目表、游戏厂商表。
2. 将原 GameMoveTest 数据迁移到当前模块，并遵守 `types/contract.ts` 中的 `GameMove` 契约。
3. 对外提供 `GameMove` API，供其他模块消费标准化游戏动态。
4. 在 `/radar` 页面展示今日摘要、库表统计、情报流周期统计和完整库表内容。
5. 支持按时间窗口、关键词和重要度查看情报流。
6. 支持三张库表在同一模块内切换查看，并提供 10 / 20 / 50 条分页。
7. 支持手动抓取全网新动态，并在服务端每天 0 点自动刷新一次。
8. 支持通过 Wikipedia / Wikidata / 公开网页补齐重点游戏和厂商信息。

## 3. 用户与场景

### 3.1 厂商运营

每天打开页面查看今日摘要和近期情报，判断哪些游戏或厂商需要优先跟进。

### 3.2 情报分析人员

查看底层库表，确认游戏、厂商、动态数据是否完整、准确、可追溯。

### 3.3 下游模块开发者

通过 `/radar/api` 获取符合 `GameMove` 契约的数据，不需要理解内部库表结构。

## 4. 页面结构

当前页面地址：

```text
/radar
```

页面由四个部分组成。

### 4.1 顶部标题、搜索和手动刷新

顶部展示模块名称、模块定位、搜索框、时间窗口筛选、重要度筛选和手动刷新按钮。

搜索支持：

- 游戏名称
- 厂商名称
- 品类
- 来源
- 摘要关键词
- 地区

时间窗口支持：

- 近 30 天，默认选中
- 近 7 天
- 近 24 小时
- 全部

重要度支持：

- 全部重要度
- 重要度至少 4
- 重要度至少 3
- 重要度至少 2

手动刷新按钮文案：

```text
手动抓取全网新动态
```

点击后调用：

```text
POST /radar/api/refresh
```

完成后刷新当前页面数据。

### 4.2 今日摘要

今日摘要按当前库表里的最新动态日期计算，不依赖客户端当前时间，避免本地调试时时区或日期不一致导致数据为空。

今日摘要展示：

- 今日新增动态数
- 今日涉及游戏数
- 今日涉及厂商数
- 重要摘要 Top 5
- 历史摘要入口

交互要求：

- 点击「今日新增动态」数字，可定位到重要摘要列表。
- 点击「涉及游戏」数字，可展开今天涉及的游戏名称列表。
- 点击「涉及厂商」数字，可展开今天涉及的厂商名称列表。
- 点击「查看历史摘要」，进入历史摘要页面。

历史摘要页面地址：

```text
/radar/summaries
```

历史摘要按日期聚合，展示每天的摘要总数、高重要度数量和当天全部摘要列表。

### 4.3 库表统计卡片

今日摘要下方保留三张整体统计卡片：

- 厂商表：展示当前厂商总数。
- 游戏项目表：展示当前游戏项目总数。
- 情报流：展示当前动态总数，以及所选时间周期内的情报数量。

情报流统计卡片支持快速切换时间周期：

- 近 30 天
- 近 7 天
- 近 24 小时
- 全部

当前周期数量必须与底层情报数据一致，不能只按页面展示条数计算。

### 4.4 库表详情查看

页面下半部分是一个详细库表查看模块，支持在三张库表之间切换：

- 游戏动态表
- 游戏项目表
- 游戏厂商表

展示规则：

- 使用列表式表格展示，不使用卡片式展示。
- 不展示内部 `id` 字段。
- 不提供「看雷达」「去诊断」「看 GIP 消耗」等跳转动作。
- 不展示字段映射模块。
- 不展示按事件类型筛选游戏动作模块。
- 支持实时搜索表内容。
- 支持手动刷新表内容。
- 每 15 秒自动刷新表格快照。
- 支持 10 / 20 / 50 条分页。
- 支持首页、上一页、下一页、末页。

## 5. 三张库表展示字段

### 5.1 游戏动态表

游戏动态表合并展示底层动态数据和 `GameMove` 契约输出结果。

页面展示字段：

| 字段 | 说明 |
|---|---|
| 日期 | 动态日期 |
| 动作 | 对应 `GameMove.moveType` |
| 游戏/厂商 | 关联游戏名称；无游戏时展示厂商或动态名称 |
| 公司 | 关联厂商名称 |
| 品类 | 游戏品类；厂商动态展示为厂商动态 |
| 摘要 | 动态摘要 |
| 来源 | 来源名称，可点击跳转到来源页面 |
| 重要度 | 1-5 分 |

说明：

- 「动作」即 `GameMove` 契约里的 `moveType`。
- 「来源」必须优先展示来源名称，并支持点击打开 `sourceUrl`。
- 页面不展示动态 `id`。

### 5.2 游戏项目表

页面展示字段：

| 字段 | 说明 |
|---|---|
| 游戏 | 游戏名称 |
| 游戏厂商 | 发行商优先；无发行商时使用开发商；再无则使用厂商表名称 |
| 阶段 | 游戏当前阶段 |
| 品类 | 游戏品类 |
| 平台 | 支持平台 |
| 区域 | 发布区域 |
| 官网 | 官方网站，可点击 |
| Wikipedia | Wikipedia 页面，可点击 |
| 最新进展 | 当前最新进展 |
| 关注度 | 内部关注度分值 |

说明：

- 发行商和开发商在页面合并为「游戏厂商」一个字段。
- 如果 Wikipedia 或公开网页存在发行商信息，优先以发行商作为游戏厂商。
- 如果没有发行商，则使用开发商。
- `releaseDate` 仍可保留在数据库中，用于判断游戏是否已经 release，但页面只展示「阶段」，不单独展示 Release 字段。
- 页面不展示游戏 `id`。

### 5.3 游戏厂商表

页面展示字段：

| 字段 | 说明 |
|---|---|
| 厂商 | 公司名称 |
| 区域/国家地区 | 合并展示区域和主要国家/地区 |
| 描述 | 公司描述 |
| 官网 | 官方网站，可点击 |
| Wikipedia | Wikipedia 页面，可点击 |
| 更新时间 | 只展示到年-月-日 |

说明：

- 厂商别名字段不在页面展示。
- 区域和主要国家/地区合并为一个展示字段。
- 更新时间只展示 `YYYY-MM-DD`，不展示更细时间。
- 页面不展示厂商 `id`。

## 7. GameMove 契约输出

本模块对外输出项目统一契约：

```ts
interface GameMove {
  gameId: string;
  name: string;
  category: string;
  moveType: "新游" | "版本更新" | "大版本" | "活动";
  summary: string;
  source: string;
  date: string;
}
```

契约映射在服务端执行，不在页面展示字段映射模块。

映射规则：

| GameMove 字段 | 来源规则 |
|---|---|
| `gameId` | 优先使用 `games.id`；厂商动态使用 `company-{companyId}`；无法匹配时使用 `update-{updateId}` |
| `name` | 优先使用游戏名称；无游戏时使用厂商名称；兜底使用动态摘要 |
| `category` | 优先使用游戏首个品类；无游戏时使用 `厂商动态` |
| `moveType` | 根据动态内容和游戏阶段推断为 `新游`、`版本更新`、`大版本`、`活动` |
| `summary` | 使用动态摘要；必要时兜底游戏最新进展或厂商描述 |
| `source` | 使用来源名称；必要时兜底发现来源或厂商名称 |
| `date` | 使用动态日期；必要时兜底游戏预计上线日期或更新时间 |

## 8. 刷新与采集

### 8.1 手动刷新

页面支持手动触发全网动态抓取：

```text
POST /radar/api/refresh
```

返回：

```ts
{
  inserted: number;
  skipped: number;
  date: string;
  sources: string[];
}
```

当前 RSS 来源包括：

- GamesIndustry.biz
- PocketGamer.biz
- Gematsu
- Game Developer

刷新流程：

1. 拉取 RSS 源。
2. 解析标题、链接、描述和发布时间。
3. 使用内容哈希去重。
4. 基于已有游戏名、游戏别名、厂商名、厂商别名匹配实体。
5. 推断动作类型和重要度。
6. 写入 `radar-data.json`。
7. 页面刷新后重新展示最新摘要、统计和库表。

### 8.2 自动刷新

服务端启动后会注册轻量定时器：

- 时区：`Asia/Shanghai`
- 时间：每天 `00:00`
- 频率检查：每 60 秒检查一次

每天 0 点只触发一次 `runRadarRefresh()`，避免重复写入。

### 8.3 表格自动刷新

库表详情模块每 15 秒请求一次：

```text
GET /radar/api/tables
```

用于保持页面内库表内容接近实时。

## 9. 数据补齐策略

### 9.1 Wikipedia / Wikidata

重点游戏和厂商信息优先从 Wikipedia / Wikidata / 公开网页补齐。

游戏厂商规则：

1. 如果能找到发行商，优先记录发行商。
2. 如果没有发行商，则记录开发商。
3. 如果都没有，则保留现有厂商表关联。

阶段规则：

1. 使用 `releaseDate` 和公开资料校验游戏是否已经上线。
2. 已上线游戏的阶段应展示为正式上线或相近表达。
3. 页面只展示阶段，不展示 release 字段。

来源规则：

- Wikipedia references 和公开报道可作为游戏动态来源。
- 游戏数据以最新日期的公开报道和网页信息为准。

### 9.2 已补齐重点样本

当前已针对以下游戏补齐公开信息：

- Blood Strike
- 燕云十六声 / Where Winds Meet
- 蛋仔派对 / Eggy Party
- 七日世界 / Once Human

## 10. API

### 10.1 GameMove API

```text
GET /radar/api
```

返回：

```ts
{
  data: GameMove[];
  total: number;
  contract: "GameMove";
}
```

支持参数：

| 参数 | 说明 |
|---|---|
| `q` | 关键词搜索 |
| `type` | `新游` / `版本更新` / `大版本` / `活动` |
| `gameId` | 指定游戏 |
| `window` | `24h` / `7d` / `30d` / `all` |
| `importance` | 最低重要度 |

示例：

```text
/radar/api
/radar/api?q=NetEase
/radar/api?type=新游
/radar/api?window=7d&importance=4
```

### 10.2 库表 API

```text
GET /radar/api/tables
```

不传 `table` 时返回完整快照：

- `companies`
- `games`
- `updates`
- `gameMoves`
- `stats`

传 `table` 时返回单表：

```text
/radar/api/tables?table=updates
/radar/api/tables?table=games
/radar/api/tables?table=companies
```

支持参数：

| 参数 | 说明 |
|---|---|
| `table` | `updates` / `games` / `companies` |
| `q` | 表内容关键词搜索 |
| `gameId` | 按游戏过滤相关情报 |

### 10.3 刷新 API

```text
POST /radar/api/refresh
```

用于手动抓取全网新动态。

## 11. 排序与筛选

### 11.1 情报流排序

情报按以下规则排序：

1. 重要度降序。
2. 同重要度按日期倒序。

### 11.2 今日摘要

今日摘要使用当前库中最新动态日期作为今日日期，并在当天情报中取重要度最高的 5 条。

### 11.3 时间窗口

时间窗口以当前库中最新动态日期作为基准计算：

- `24h`：最新日期往前 1 天。
- `7d`：最新日期往前 7 天。
- `30d`：最新日期往前 30 天。
- `all`：全部情报。

这样可以保证本地调试时，即使系统日期与样本数据日期不同，周期统计仍然准确。
