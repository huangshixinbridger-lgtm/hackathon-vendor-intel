# PRD · 模块① 情报抓取

> 版本：v1.0  
> 所属项目：「情报与运营台」TikTok Gaming 厂商运营提效平台  
> 模块 Owner：Gardner  
> 输入依据：新版全项目 PRD + 原「全球游戏厂商动态追踪工具 PRD」  

## 1. 模块定位

情报抓取模块是「情报与运营台」的发现入口，负责自动追踪全球游戏厂商、游戏项目和游戏动态，回答厂商运营每天最关心的问题：

> 最近哪些游戏或厂商有值得关注的市场动作？我今天该优先看谁？

本模块不做 TikTok 内容生态分析，不判断内容梗、自来水、达人结构等机会信号；这些属于「内容雷达」模块。本模块只聚焦厂商/市场动作，例如新游上线、版本更新、电竞赛事、IP 联动、预注册、收入表现、停服、融资、收购等。

## 2. 产品目标

1. 建立游戏厂商、游戏项目、游戏动态三张核心数据库表。
2. 支持从媒体、公告、RSS、中文游戏媒体等来源采集动态。
3. 支持历史回溯，把已有数据库中的公司、游戏、动态完整沉淀。
4. 将原始动态统一映射成项目契约 `GameMove`，供其他模块调用。
5. 在页面中实时查看公司表、游戏表、动态表和 `GameMove` 输出。
6. 让运营可以按时间窗、重要度、品类、厂商、地区、关键词筛选情报。
7. 每条情报提供重要度、摘要、来源、关联游戏/厂商，以及「对运营意味着什么」。
8. 每条情报可跳转到内容雷达、诊断报告、GIP 看板。

## 3. 用户角色

### 3.1 厂商运营

每天巡逻情报流，快速发现近期有动作的游戏或厂商，并决定是否进入诊断、预算查询或内容分析。

### 3.2 情报分析人员

维护游戏厂商和游戏项目数据库，关注采集质量、实体匹配、漏收录、误收录和来源可靠性。

### 3.3 管理者

查看每日重点摘要，了解市场变化和潜在 GIP 机会。

## 4. 核心用户故事

1. 作为厂商运营，我每天打开情报流，可以快速看到最近 7 天有哪些新游、大版本、联动、电竞赛事或营销动作。
2. 作为厂商运营，我可以按自己负责的品类、厂商、地区筛选，只看和我相关的情报。
3. 作为厂商运营，我希望每条情报不只是新闻摘要，还能告诉我「这对运营意味着什么」。
4. 作为厂商运营，我看到重要情报后，可以一键跳转到内容雷达、诊断报告或 GIP 看板。
5. 作为情报分析人员，我可以查看原始库表，确认公司、游戏、动态是否正确入库。
6. 作为情报分析人员，我可以通过 API 给其他模块提供标准化 `GameMove` 数据。

### 5.4 契约输出 `GameMove`

本模块必须输出当前项目统一契约：

```ts
interface GameMove {
  gameId: string;
  name: string;
  category: string;
  moveType: "新游" | "版本更新" | "大版本" | "活动";
  summary: string;
  source: string;
  date: string;
}
```

字段映射：

| GameMove 字段 | 来源 |
|---|---|
| `gameId` | `games.id`；若无游戏，仅有厂商动态，则用 `company-{companyId}` |
| `name` | `games.name`；若无游戏，则用 `companies.name` |
| `category` | `games.genres[0]`；若无游戏，则为 `厂商动态` |
| `moveType` | 根据 `updates.summary/detail`、`games.stage/latestProgress` 推断 |
| `summary` | `updates.summary` |
| `source` | `updates.sourceName` |
| `date` | `updates.updateDate` |

## 6. 事件类型

产品层事件类型包括：

- 新游上线
- 大版本更新
- 电竞赛事
- IP 联动
- 预注册开启
- 营销大动作
- 收入/下载/销量变化
- 融资/收购/裁员/组织调整
- 停服/停止服务
- 其他

对外输出到 `GameMove.moveType` 时，需要收敛成契约允许的四类：

- `新游`
- `版本更新`
- `大版本`
- `活动`

## 7. 收录规则

### 7.1 已有游戏和厂商优先

每日采集时，优先匹配数据库里已有的游戏、公司和别名。命中已有实体的动态优先保留。

### 7.2 新游戏收录门槛

新游戏需要满足以下条件之一或多个：

- 来自主流游戏媒体或行业媒体；
- 标题或摘要包含现代游戏平台信号；
- 包含上线、测试、预约、软启动、收入、下载、销量、融资、收购、停服等关注信号；
- 命中重点 IP、重点厂商或高关注项目关键词；
- 可从标题中抽取出合理游戏名。

### 7.3 老游戏收录门槛

老游戏仅在出现重大信号时保留：

- 现代平台上线或更新；
- 收入、下载、销量、活跃表现；
- 停服或停止服务；
- 重大版本、资料片、跨区发行；
- 重新发行或重要平台迁移。

### 7.4 数据规模

游戏库上限为 3000 款。达到上限后，不再自动发现新游戏，但继续更新已有游戏和厂商动态。

## 8. 采集能力

### 8.1 每日自动更新

系统按配置时区和时间每日执行一次采集。

默认：

- 时区：`Asia/Shanghai`
- 时间：每天 0 点

流程：

1. 拉取 RSS、中文媒体、公告或配置源。
2. 按已有公司、游戏、别名做实体匹配。
3. 按平台和关注度规则过滤。
4. 推断阶段、区域、品类、平台和事件类型。
5. 写入 `updates` 表。
6. 必要时自动发现并写入新游戏。
7. 更新游戏最新进展。
8. 生成每日重点摘要。
9. 如启用飞书，则同步到飞书多维表格。

### 8.2 手动立即更新

页面提供「立即更新」入口。执行完成后刷新：

- 数据量统计
- 情报流
- 原始表格
- 摘要

### 8.3 历史回溯

支持回溯过去 12 个月数据，写入：

- 公司表
- 游戏表
- 动态表
- 摘要文件

### 8.4 中文渠道

支持国内游戏媒体、自媒体、官方账号来源，例如：

- 游戏葡萄
- 游研社
- 游戏茶馆
- 微信 RSSHub
- 小红书 RSSHub
- 腾讯新闻游戏频道

微信、小红书等需要登录态的渠道，不在本期硬编码 Cookie。

## 9. 页面功能

### 9.1 情报流

情报流按重要度降序，同级按时间倒序。

每张情报卡包含：

- 事件类型标签
- 关联游戏/厂商
- 品类
- 时间
- 来源
- 2-3 句摘要
- 对运营意味着什么
- 重要度
- 操作入口：看雷达 / 出诊断 / 查 GIP

### 9.2 今日摘要

顶部展示 AI 或规则生成的今日重点，并支持查看全部摘要和历史摘要：

- 今天值得关注的 3 件事
- 当前时间窗内的全部摘要信息
- 历史摘要次级页面
- 新增动态数
- 高重要度动态数
- 涉及重点厂商/游戏

### 9.3 筛选栏

支持：

- 时间窗：24h / 7d / 30d / 自定义
- 品类
- 厂商
- 地区
- 关键词
- 重要度

### 9.4 数据表查看

页面支持查看三类表：

- 游戏项目表
- 游戏动态表
- 游戏厂商表

其中游戏动态表合并展示原动态表和 `GameMove` 契约输出字段，来源可点击跳转到原始页面。支持搜索、刷新和自动刷新。

## 10. API

### 10.1 给下游模块使用

```text
GET /radar/api
```

返回 `GameMove[]`。

支持参数：

```text
q
type
gameId
```

示例：

```text
/radar/api
/radar/api?type=新游
/radar/api?q=NetEase
/radar/api?gameId=123
```

### 10.2 给页面表格使用

```text
GET /radar/api/tables
```

返回完整快照：

- companies
- games
- updates
- gameMoves
- stats
- mapping

支持参数：

```text
table=companies | games | updates
q=关键词
gameId=游戏ID
```

示例：

```text
/radar/api/tables
/radar/api/tables?table=games&q=RPG
/radar/api/tables?table=updates
```

## 11. 与其他模块联动

### 11.1 到内容雷达

情报卡点击「看雷达」，传递 `gameId`。实际落点以后续模块命名为准。

### 11.2 到诊断报告

情报卡点击「出诊断」，传递 `gameId`：

```text
/diagnosis?gameId=xxx
```

诊断模块需要消费本模块的近期情报。

### 11.3 到 GIP 看板

情报卡点击「查 GIP」，传递 `gameId`：

```text
/gip?gameId=xxx
```

GIP 看板的时间线事件应引用本模块情报数据。

## 12. 非目标

本期不做：

- TikTok 内容生态分析；
- 梗、热音、自来水、达人结构判断；
- 自动执行营销动作；
- 权限体系；
- 复杂审核流；
- 大规模分布式采集；
- 付费榜单全量接入；
- 微信/小红书账号自动登录抓取；
- 游戏实体人工合并拆分 UI。

## 13. 验收标准

1. 页面能展示真实导入的公司、游戏、动态数据。
2. `GameMove` API 能返回符合 `types/contract.ts` 的数据。
3. 筛选、搜索、排序可用。
4. 每条情报有来源、时间、摘要、重要度。
5. 动态能正确映射到游戏；无游戏动态能兜底映射到厂商。
6. 情报卡三个跳转入口正确。
7. 数据统计准确展示公司数、游戏数、动态数、`GameMove` 数。
8. 不出现 hydration mismatch 等运行时报错。
9. 与诊断、GIP 模块通过 `gameId` 联动。
10. 本地 `typecheck` 和 `build` 通过。

## 14. 相对原单模块 PRD 的改动

1. 用户范围从独立工具的多角色，收敛为以厂商运营为核心用户。
2. 产品定位从独立追踪工具，调整为全项目链路中的发现入口。
3. 新增 `GameMove` 契约输出，供诊断、GIP 和其他模块调用。
4. 新增三张旧库表到 `GameMove` 的字段映射规则。
5. 新增「对运营意味着什么」，把新闻摘要翻译成运营动作判断。
6. 新增情报流结构：今日摘要、筛选栏、重要度排序、卡片操作。
7. 新增模块联动：看雷达、出诊断、查 GIP。
8. 明确边界：不做 TikTok 内容生态判断，这部分交给内容雷达。
9. API 从旧工具的 Flask 风格，调整为当前 Next.js 模块路径。
10. 保留原 PRD 的数据库、采集、回溯、中文渠道、飞书同步和摘要能力。
